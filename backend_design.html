1. Overview & requirements recap

  • Pools are created by the user with a fixed capacity (max number of tickets).
  • Tickets are added by pasting a Jira issue URL; the backend resolves it via the Jira API and stores title, description, and deadline.
  • Deadline warnings fire on a per-user configurable lead time, up to 2 hours before the deadline.
  • Capacity guard: if adding a ticket would exceed pool capacity, the request is rejected until the user removes an existing ticket.
  • Overdue state: once a deadline passes, a continuous (repeating, not one-shot) red warning is shown until the ticket is removed or its deadline is updated.
  • Clients run on Windows, macOS, and mobile — the backend is a single multi-tenant service exposed over REST + a persistent real-time channel, so state stays in sync across a user's devices.
"Nouns" in the original brief is treated here as a placeholder for a generic external notification/chat tool (Slack/Teams-style). The integration layer is built as a pluggable provider interface so any such tool — or Jira itself as a notification target — can be wired in without touching core logic.

SQLite is a single-writer, single-file database — great for a self-hosted/single-instance backend (e.g. one server per team) with low ops overhead. It does not support running many API instances against the same DB file over a network. If you later need multiple horizontally-scaled API instances, plan to migrate to libSQL/Turso (SQLite-compatible, network-accessible) or Postgres — the Prisma layer makes that swap mostly a config change.

5. Database schema (SQLite DDL)

SQLite has no native UUID/TIMESTAMPTZ/BYTEA types — ids are stored as TEXT (app-generated UUIDs), timestamps as ISO-8601 TEXT, and encrypted tokens as BLOB. Foreign keys must be turned on explicitly (PRAGMA foreign_keys = ON;). Shown here as raw SQL; in practice this is generated from a Prisma schema.

PRAGMA foreign_keys = ON;

CREATE TABLE users (
  id TEXT PRIMARY KEY,                 -- uuid, generated in app code
  email TEXT UNIQUE NOT NULL,
  display_name TEXT NOT NULL,
  default_warning_lead_minutes INTEGER NOT NULL DEFAULT 60
    CHECK (default_warning_lead_minutes BETWEEN 1 AND 120),
  created_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now'))
);

CREATE TABLE pools (
  id TEXT PRIMARY KEY,
  owner_id TEXT NOT NULL REFERENCES users(id),
  name TEXT NOT NULL,
  capacity INTEGER NOT NULL CHECK (capacity > 0),
  created_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now'))
);

CREATE TABLE tickets (
  id TEXT PRIMARY KEY,
  pool_id TEXT NOT NULL REFERENCES pools(id) ON DELETE CASCADE,
  jira_key TEXT NOT NULL,
  jira_url TEXT NOT NULL,
  title TEXT NOT NULL,
  description TEXT,
  deadline TEXT NOT NULL,               -- ISO-8601 UTC
  warning_lead_minutes INTEGER CHECK (warning_lead_minutes BETWEEN 1 AND 120),
  status TEXT NOT NULL DEFAULT 'Normal', -- Normal | Warning | Overdue
  added_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
  UNIQUE (pool_id, jira_key)
);
CREATE INDEX idx_tickets_deadline ON tickets (deadline);

CREATE TABLE integration_connections (
  id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL REFERENCES users(id),
  provider TEXT NOT NULL,               -- 'jira' | 'notifier'
  auth_token_encrypted BLOB NOT NULL,
  refresh_token_encrypted BLOB,
  expires_at TEXT,
  UNIQUE (user_id, provider)
);

CREATE TABLE notification_events (
  id TEXT PRIMARY KEY,
  ticket_id TEXT NOT NULL REFERENCES tickets(id) ON DELETE CASCADE,
  type TEXT NOT NULL,                   -- Warning | Overdue | CapacityBlocked
  sent_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
  channel TEXT NOT NULL                 -- 'push' | 'chat' | 'email'
);

Note: SQLite allows only one writer at a time. Wrap the capacity-check-then-insert in BEGIN IMMEDIATE ... COMMIT (via Prisma's interactive transactions) so concurrent add-ticket requests serialize instead of racing.

Add-ticket flow (capacity check)

POST /pools/{id}/tickets
{ "jiraUrl": "https://yourorg.atlassian.net/browse/ENG-482" }

1. Parse issue key (ENG-482) from URL.
2. If pool.ticketCount >= pool.capacity -> 409 Conflict
   { "error": "PoolCapacityExceeded", "capacity": 10, "current": 10 }
3. Else call Jira API GET /rest/api/3/issue/ENG-482
4. Map summary/description/duedate -> Ticket, insert inside a SQLite
   BEGIN IMMEDIATE transaction that re-checks count (guards races from
   two devices adding at once, since SQLite serializes writers anyway).
5. Broadcast TicketAdded over Socket.IO to all of the user's connected clients.

7. Real-time sync (Socket.IO)

Each authenticated client opens a Socket.IO connection and joins a room keyed by userId (or poolId). Server-pushed events:

EventPayloadTrigger
TicketWarningticketId, deadline, minutesRemainingBackground scan crosses the lead-time threshold
TicketOverdueticketId, deadlineRepeats every N seconds/minutes while the ticket remains in the pool and unresolved — drives the continuous red banner
TicketResolvedticketIdTicket removed or deadline pushed forward; clears the warning/overdue UI
PoolCapacityChangedpoolId, current, capacityTicket added/removed, so all devices' "add" button enables/disables consistently

8. Deadline warning engine

A node-cron job runs in-process every 30–60 seconds and evaluates every active ticket:

cron.schedule('*/30 * * * * *', async () => {
  const tickets = await prisma.ticket.findMany({ where: { status: { not: 'Resolved' } } });
  const now = new Date();

  for (const ticket of tickets) {
    const lead = ticket.warningLeadMinutes ?? ticket.pool.owner.defaultWarningLeadMinutes; // 1-120 min

    if (now >= ticket.deadline) {
      if (ticket.status !== 'Overdue') {
        await prisma.ticket.update({ where: { id: ticket.id }, data: { status: 'Overdue' } });
        io.to(ticket.poolId).emit('TicketOverdue', { ticketId: ticket.id, deadline: ticket.deadline }); // first fire
      } else {
        io.to(ticket.poolId).emit('TicketOverdue', { ticketId: ticket.id, deadline: ticket.deadline }); // repeat -> continuous red banner
      }
    } else if (now >= new Date(ticket.deadline.getTime() - lead * 60000)) {
      if (ticket.status !== 'Warning') {
        await prisma.ticket.update({ where: { id: ticket.id }, data: { status: 'Warning' } });
        io.to(ticket.poolId).emit('TicketWarning', { ticketId: ticket.id, deadline: ticket.deadline });
      }
    }
    // else: status stays 'Normal', nothing to do
  }
});

The "continuous" overdue warning is implemented as a repeating push rather than a one-shot event — the client renders it as a persistent red banner as long as it keeps receiving/holding that state, and clears it only on TicketResolved.

9. Pool capacity enforcement

  • Capacity check happens server-side inside the same DB transaction as the insert (not just in the client), so two devices adding simultaneously can't both slip past a full pool.
  • Rejection response is explicit (PoolCapacityExceeded) so the client can prompt "remove a ticket first" rather than silently failing.
  • Capacity changes (PATCH /pools/{id}) are allowed at any time; shrinking capacity below the current ticket count is accepted but blocks further additions until the user is back under the new limit.

10. External integrations

Jira

  • OAuth 2.0 (3LO) per user; tokens stored encrypted in integration_connections.
  • Issue resolution: parse the pasted URL for the issue key, call GET /rest/api/3/issue/{key}, map fields.summary, fields.description, fields.duedate → ticket.
  • Optional: subscribe to Jira webhooks (issue_updated) to keep the stored deadline in sync if it changes on the Jira side, instead of relying purely on polling.

Generic notification provider (the "Nouns" slot)

Modeled as a pluggable interface so any chat/notification tool can be wired in without touching pool/ticket logic:

export interface NotificationProvider {
  readonly providerName: string;
  send(event: NotificationEvent, recipient: User): Promise<void>;
}

// Implementations: SlackNotifier, TeamsNotifier, PushNotifier, EmailNotifier, ...
// Resolved per user at runtime from their IntegrationConnection.provider field,
// e.g. via a small factory/registry keyed by provider name.

This keeps the door open for whichever tool "Nouns" actually refers to — swap in a concrete implementation without changing the domain or API layer.

11. Security, scaling, offline

ConcernApproach
AuthJWT access + refresh tokens for app users; OAuth2 for Jira/notifier, tokens encrypted at rest (Node crypto/libsodium, key from env/secrets manager)
ScalingSingle Node process owns the SQLite file (one instance per deployment/team). This is intentionally simple — no backplane needed since there's only one process to broadcast from. If you outgrow it, move to libSQL/Turso or Postgres and add a Socket.IO Redis adapter for multi-instance fan-out
Jira rate limitsCache issue lookups briefly in-memory (Node Map with TTL); batch/backoff on 429s; webhook-driven updates reduce polling
Offline clientsOn reconnect, client calls GET /pools/{id}/tickets to resync full state, then resumes the Socket.IO stream for deltas
Multi-device consistencyAll writes go through the same API + DB; Socket.IO room broadcast keeps Windows/macOS/mobile clients visually identical within ~1s
BackupsSQLite is a single file — back it up with simple file-copy/snapshot on an interval (or use litestream for continuous replication to S3-compatible storage)
Input validationStrict Jira URL/key regex validation before calling out to Jira, to avoid SSRF via arbitrary URLs
